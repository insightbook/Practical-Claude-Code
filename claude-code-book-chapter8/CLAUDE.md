# 프로젝트 메모리

## 기술 스택

- 개발 환경: devcontainer
- Node.js v24.11.0
- TypeScript 5.x
- 패키지 매니저: npm

## 스펙 주도 개발 기본 원칙

### 기본 흐름

1. **문서 생성**: 영구 문서(`docs/`)로 "무엇을 만들 것인가"를 정의
2. **작업 계획**: 스티어링 파일(`.steering/`)로 "이번에 무엇을 할 것인가"를 계획
3. **구현**: tasklist.md에 따라 구현하고 진행 상황을 수시로 업데이트
4. **검증**: 테스트와 동작 확인
5. **업데이트**: 필요에 따라 문서 업데이트

### 중요한 규칙

#### 문서 생성

**파일 하나를 생성한 후 반드시 확인을 받고 다음 파일 생성을 진행한다.**

확인 대기 상태를 명확하게 전달한다:
```
[문서명] 생성이 완료되었습니다. 내용을 확인해 주십시오.
확인하면 다음 문서로 진행합니다.
```

#### 구현 전 확인

새로운 구현을 시작하기 전에 반드시 다음을 확인:

1. CLAUDE.md를 읽는다
2. 관련된 영구 문서(`docs/`)를 읽는다
3. Grep으로 기존의 유사한 구현 패턴을 검색
4. 기존 패턴을 이해하고 나서 구현 개시

#### 스티어링 파일 관리

작업마다 `.steering/[YYYYMMDD]-[작업명]/` 생성:

- `requirements.md`: 이번 요구 내용
- `design.md`: 구현 접근 방식
- `tasklist.md`: 구체적인 작업 목록

명명 규칙: `20250115-add-user-profile` 형식

#### 스티어링 파일 관리

**작업 계획, 구현, 검증 시 `steering` 스킬을 사용한다.**

- **작업 계획 시**: `Skill('steering')`으로 모드 1(스티어링 파일 생성)
- **구현 시**: `Skill('steering')`으로 모드 2(구현과 tasklist.md 업데이트 관리)
- **검증 시**: `Skill('steering')`으로 모드 3(회고)

상세한 작업 순서와 업데이트 관리 관련 규칙은 steering 스킬에 정의되어 있습니다.

## 디렉터리 구조

### 영구 문서 (`docs/`)

애플리케이션 전체의 "무엇을 만들 것인가", "어떻게 만들 것인가"를 정의:

#### 초안, 아이디어 (`docs/ideas/`)
- 아이디어 상담, 브레인스토밍의 성과물
- 기술 조사 메모
- 자유 형식 (최소한의 구조화)
- `/setup-project` 실행 시 자동으로 로드됨

#### 정식판 문서
- **product-requirements.md** - 제품 요구사항 명세서
- **functional-design.md** - 기능 설계서
- **architecture.md** - 기술 사양서
- **repository-structure.md** - 리포지토리 구조 정의서
- **development-guidelines.md** - 개발 가이드라인
- **glossary.md** - 보편 언어 정의

### 작업 단위 문서 (`.steering/`)

특정 개발 작업에 있어서 "이번에 무엇을 할 것인가"를 정의:

- `requirements.md`: 이번 작업의 요구 내용
- `design.md`: 변경 내용 설계
- `tasklist.md`: 작업 목록

## 개발 프로세스

### 초기 설정

1. 이 템플릿을 사용
2. `/setup-project` 로 영구 문서 생성(대화형으로 6개 생성)
3. `/add-feature [기능]` 으로 기능 구현

### 일반적인 사용법

**평범하게 대화를 통해 작업을 요청합니다.**

```bash
# 문서 편집
> PRD에 신기능을 추가해 주세요
> architecture.md의 성능 요구사항을 재검토해
> glossary.md에 새로운 도메인 용어를 추가해

# 기능 추가(정해진 작업 흐름)
> /add-feature 유저 프로필 편집

# 상세 리뷰(상세한 보고서가 필요할 때)
> /review-docs docs/product-requirements.md
```

**중요**: 스펙 주도 개발의 세부적인 내용은 의식할 필요가 없습니다. Claude Code가 적절한 스킬을 판단하여 로드합니다.

## 문서 관리 원칙

### 영구 문서 (`docs/`)

- 기본 설계를 기술
- 빈번하게 업데이트되지 않음
- 프로젝트 전체의 기준 문서로 기능

### 작업 단위 문서 (`.steering/`)

- 특정 작업에 특화
- 작업마다 신규 생성
- 이력으로서 유지
